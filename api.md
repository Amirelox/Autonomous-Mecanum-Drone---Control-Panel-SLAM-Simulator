Układ Współrzędnych i Poziomy Sterowania2.2 Układ współrzędnych (Coordinate system)Pojazd porusza się w układzie trzech osi swobody (planarnej bazy Mecanum):Pole (Field)Znaczenie (Meaning)Dodatni kierunek (Positive direction)vyJazda przód / tyłPrzód (forward)vxRuch boczny (strafe)Prawo (right)wObrót w miejscu (yaw)Zgodnie z ruchem wskazówek zegara (clockwise)💡 Przykład miksowania: Wysłanie ramki {"vx": 0.3, "vy": 0.5, "w": -0.2} oznacza: „jedź do przodu i w prawo, jednocześnie lekko skręcając w lewo”.2.3 Dwa poziomy sterowania (Two command levels)Poziom 2 (body commands): Podajesz docelowy wektor ruchu ciała (vx, vy, w lub nazwany ruch). Firmware automatycznie miksuje go na prędkości 4 kół. Jest to rekomendowany poziom do normalnej jazdy.Poziom 1 (raw wheels): Sterujesz bezpośrednio każdym kołem z osobna za pomocą tablicy: [FL, FR, RL, RR] (kolejno: front-left, front-right, rear-left, rear-right). Pomija mikser. Używany wyłącznie do testów i diagnostyki silników.📐 Dla ciekawskich: Mikser Poziomu 2 realizuje następujące równania kinematyczne:$$FL = vy + vx + w$$$$FR = vy - vx - w$$$$RL = vy - vx + w$$$$RR = vy + vx - w$$Wynik jest następnie skalowany w dół, aby żadne z kół nie przekroczyło wartości $1.0$. Pozwala to zachować właściwy kierunek ruchu nawet przy maksymalnym wychyleniu drążka.3. Model Bezpieczeństwa (Safety model)Każda komenda przesyłana do robota przechodzi przez rdzeń bezpieczeństwa według następujących reguł:Watchdog (domyślnie 350 ms): Jeśli w tym czasie nie dotrze żadna nowa komenda, robot zatrzymuje się i rozbraja (disarms). Podczas ruchu należy bezustannie strumieniować pakiety z częstotliwością 20–50 Hz. Pojedyncza komenda przemieszcza robota na maksymalnie 0.35 sekundy.Uzbrajanie (Arming): Po uruchomieniu, zatrzymaniu awaryjnym (E-stop), zadziałaniu watchdog-a lub przełączeniu między WiFi a UART, robot jest rozbrojony. Uzbroi się ponownie tylko wtedy, gdy otrzyma komendę neutralną (zbliżoną do 0). Chroni to robota przed gwałtownym skokiem do przodu w momencie połączenia.E-stop: Wysłanie {"cmd": "estop"} natychmiast wyłącza silniki i blokuje jazdę. Komenda {"cmd": "resume"} odblokowuje układ. Następnie należy ponownie uzbroić robota komendą neutralną.Zasada ostatniej komendy: Robot wykonuje zawsze najnowszą poprawną komendę (brak kolejkowania).Clamp + walidacja: Wartości wejściowe są obcinane do zakresu $[-1 \dots +1]$. Komendy zwierające wartość NaN są odrzucane.Slew limit (domyślnie 4.0/s): Zmiany prędkości kół są rampowane (od 0 do maksimum w ~0.25 s) w celu ograniczenia skoków prądowych i ochrony akumulatora.Deadband (domyślnie 0.03): Bardzo małe wartości wejściowe są traktowane jako 0 (filtr szumów drążków).Start floor (min_duty, domyślnie 0.55): Silniki z przekładnią utknęłyby (stall) przy zbyt niskim sygnale PWM. Każde koło, które powinno się kręcić, dostaje co najmniej taką wartość wypełnienia.4. Autentykacja (WebSocket handshake)Autoryzacja przebiega w trybie challenge-response (klucz nie jest przesyłany bezpośrednio przez sieć):Połącz się z adresem ws://<robot>/ws.Robot wysyła pakiet startowy: {"nonce": "<32 znaki hex>"} (16 losowych bajtów w formacie hex).Oblicz podpis za pomocą algorytmu HMAC-SHA256(key_bytes, nonce_bytes).⚠️ Ważne: Najpierw przekonwertuj (zdekoduj) klucz oraz nonce z formatu hex na surowe bajty.Wyślij odpowiedź: {"auth": "<64 znaki hex>"}.W przypadku sukcesu robot odpowiada: {"auth": "ok", "token": "<32 znaki hex>", "level": N}, gdzie N=1 to poziom kontroli, a N=2 to poziom administratora.Jednowierszowy ekwiwalent obliczenia podpisu w Pythonie:Pythonimport hmac, hashlib
answer = hmac.new(bytes.fromhex(KEY_HEX), bytes.fromhex(nonce_hex), hashlib.sha256).hexdigest()

Po pomyślnej autoryzacji WebSocket komunikacja ruchowa nie jest już szyfrowana, co minimalizuje opóźnienia (latency).5. Kompletny program demonstracyjny w PythoniePoniższy skrypt realizuje pełną procedurę autentykacji handshake i wykonuje pokazowy przejazd po obrysie kwadratu:Python# Wymaga instalacji: pip install websocket-client
import json, time, hmac, hashlib
import websocket

ROBOT = "ws://<robot>/ws"                 # Adres IP dostarczony przez administratora
KEY_HEX = "<twoj_64_znakowy_klucz>"       # Trzymaj klucz poza repozytorium git!

def recv_json(ws):
    return json.loads(ws.recv())

# -- 1. Połączenie i autoryzacja challenge-response -------------------------
ws = websocket.create_connection(ROBOT, timeout=5)
nonce = recv_json(ws)["nonce"]

answer = hmac.new(bytes.fromhex(KEY_HEX), bytes.fromhex(nonce), hashlib.sha256).hexdigest()
ws.send(json.dumps({"auth": answer}))

reply = recv_json(ws)
while "auth" not in reply:
    # W międzyczasie mogą napływać ramki telemetrii, ignorujemy je i szukamy odpowiedzi auth
    reply = recv_json(ws)

if reply.get("auth") != "ok":
    raise SystemExit("Błąd autoryzacji - sprawdź poprawność klucza KEY_HEX")

print("Authed, level:", reply["level"], "token:", reply["token"][:8] + "...")

# -- 2. Funkcja pomocnicza do strumieniowania komend przez dany czas ----------
def drive(msg, seconds, hz=25):
    end = time.time() + seconds
    while time.time() < end:
        ws.send(json.dumps(msg))           # Strumieniowanie karmi watchdog 350ms
        time.sleep(1.0 / hz)

# -- 3. Uzbrojenie sygnałem neutralnym, a następnie przejazd testowy ----------
drive({"vx": 0, "vy": 0, "w": 0}, 0.5)     # Wysłanie zera uzbraja robota
drive({"dir": "forward",      "spd": 0.6}, 1.0)
drive({"dir": "strafe_right", "spd": 0.6}, 1.0)
drive({"dir": "back",         "spd": 0.6}, 1.0)
drive({"dir": "strafe_left",  "spd": 0.6}, 1.0)
drive({"vx": 0.0, "vy": 0.4, "w": 0.4},  1.5)  # Ruch złożony: łuk po skosie

ws.send(json.dumps({"dir": "stop"}))       # Zawsze zatrzymaj przed rozłączeniem
ws.close()
print("Przejazd zakończony pomyślnie.")
10. Protokół UART (Komunikacja z Raspberry Pi)Drugim medium transportowym jest przewodowy interfejs UART (dedykowany dla mikrokomputera Pi zamontowanego bezpośrednio na robocie). Prędkość transmisji to 921600 baud, parametry: 8N1. Strona robota: UART1, piny TX = GP43, RX = GP44.Struktura ramki binarnej:AA 55 | TYPE(1) | SEQ(4 LE) | LEN(1) | PAYLOAD(LEN) | MAC(8 LE)
MAC: Każda ramka jest podpisana algorytmem SipHash-2-4, kluczowanym przez pierwsze 16 bajtów klucza kontrolnego robota. Ramki ze złym podpisem MAC są natychmiast odrzucane.SEQ: Numer sekwencyjny musi rosnąć z każdą ramką (ochrona przed atakiem powtórzeniowym - replay protection).Typy komunikatów (TYPE):0x01 (DRIVE_BODY): int16 wartości vx, vy, w (każda przemnożona $\times 1000$).0x02 (DRIVE_RAW): int16 $\times 4$ dla kół FL, FR, RL, RR (wartości $\times 1000$).0x03 (DRIVE_DIR): uint8 indeks kierunku ruchu + int16 prędkość spd ($\times 1000$).0x05 (CONTROL): uint8 podkomendy: 0 estop, 1 resume, 2 set_mode, 3 reboot, 4 seq_reset.0x10 / 0x11 / 0x12: Paczki danych aktualizacji firmware-u OTA przez UART.0x80 / 0x81: Ramki potwierdzenia ACK / NAK wysyłane z robota do Raspberry Pi zawierające uint32 numeru sekwencji.11. Lista Dobrych Praktyk (Best practices checklist)Pojedyncza pętla wysyłająca: Dokładnie jedno miejsce w Twoim kodzie powinno wysyłać komendy ruchu ze stałą częstotliwością 20–50 Hz. Dwa konkurujące wątki stworzą zjawisko szarpania (jitter).Najpierw uzbrój: Zawsze rozpoczynaj sesję od wysyłania przez ~0.5 s komend neutralnych (samych zer).Zatrzymaj przy wyjściu: Zawsze wyślij jawną ramkę {"dir": "stop"} przed zamknięciem socketu. Nie polegaj na tym, że watchdog zatrzyma robota za Ciebie.Konstrukcja try/finally: Owiń kod sterujący w blok try/finally, aby komenda stop została wysłana nawet wtedy, gdy Twój skrypt na komputerze ulegnie awarii.Obserwuj telemetrię: Flaga armed=0 doskonale wyjaśnia problem „dlaczego robot nie jedzie”, a rosnąca wartość age_ms wskazuje na zacinanie strumienia sieciowego.Chroń swój klucz: Nigdy nie wrzucaj klucza autoryzacji do repozytoriów Git ani dokumentów współdzielonych.12.
